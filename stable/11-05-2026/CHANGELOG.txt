REGLA DE VERSIONING (establecida 08-05-2026, reforzada 08-05-2026):
  · X0.XX es una versión del SISTEMA COMPLETO, no por archivo.
  · VERSION en cada archivo = release donde ese archivo fue tocado por última vez.
  · Cada archivo lleva dos líneas en docstring:
      Sistema: X0.XX  ← release actual del sistema (igual en todos)
      Archivo: X0.YY  ← última vez que este archivo fue modificado
  · captura_externa.py usa esquema propio v2.X (standalone) pero sus
    cambios se documentan aquí bajo el release del sistema que los acompañó.
  · Docstring y VERSION en código deben coincidir siempre.
  · CHANGELOG se actualiza ANTES de copiar archivos a producción.

REGLA DE BUMP (reforzada 08-05-2026 — para evitar micro-bumps):
  · MÁXIMO 1 bump por sesión de trabajo, sin importar cuántos fixes se hagan.
  · Durante la sesión: acumular todos los cambios, corregir todos los bugs.
  · El bump y el CHANGELOG van AL FINAL, cuando todo está validado.
  · Si un fix genera otro bug mid-sesión: corregir SIN bumpar, mismo número.
  · Ejemplo correcto: X0.68 → X0.72 en una sola entrada del CHANGELOG
    con todos los cambios listados. NO cuatro entradas X0.69/70/71/72.

VERSION X1.06 — Retry automático en feroldi_pensante.py ante errores de red — 11-05-2026
  call_pensante() ahora reintenta hasta 3 veces con backoff 10s/30s/60s cuando DeepSeek
  cierra la conexión sin responder (RemoteDisconnected, ConnectionResetError, URLError,
  HTTP 5xx). Errores HTTP 4xx (auth, bad request) no se reintentan — son del cliente.
  Razón: en producción con el segundo bot apareció "RemoteDisconnected: Remote end closed
  connection without response" que mató el pipeline en el primer intento sin ningún reintento.
  DeepSeek cae esporádicamente bajo carga; 3 reintentos cubren la mayoría de los casos.

Razón: corridas múltiples del mismo ticker en el mismo día fallaban al actualizar porque la API de GitHub requiere `sha` para overwrite. VERSION X1.01 — handler.js + HOOK.md + SOUL.md + FEROLDI.md: trigger format cambiado de `//TICKER PRECIO` (doble slash prefix) a doble-letra (cada letra del ticker repetida dos veces). Razón: test móvil de X1.00 reveló que `//` no dispara el hook desde Telegram — el cliente o el gateway interpretan `//` como prefijo de comando bot y el mensaje no llega al handler como texto plano. La doble-letra es robusta porque: (a) no usa caracteres reservados de Telegram/Markdown; (b) trivial de tipear en mobile (cada letra dos veces); (c) ningún ticker real coincide con palabra natural en español. El handler valida con `unmaskTicker()`: longitud par + cada par de caracteres idénticos; si falla la validación, el mensaje pasa al LLM sin trigger. Obfuscación bodyForAgent re-introducida (removida en X1.00) — Kika ve `N K E  P I P E L I N E` en lugar del ticker crudo para evitar análisis paralelo — 11-05-2026

    · current_assets (AssetsCurrent) y current_liabilities (LiabilitiesCurrent)
      se extraen del balance sheet XBRL — necesarios para Current Ratio real
    · stockholders_equity (StockholdersEquity con fallback a variante non-controlling)
      se extrae del balance sheet — base para D/E Ratio y ROIC
    · depreciation_amortization (DepreciationDepletionAndAmortization / Depreciation)

    · revenue_yoy_calc = (revenue - revenue_prior_year) / revenue_prior_year

[NUEVO] feroldi_normalizar.py
  Única responsabilidad: tomar datos crudos y producir JSON canónico limpio.
  Principios de diseño:
    1. Una cadena de prioridad explícita por campo (AV → EDGAR → calculado → null)
    2. Nunca crashea — siempre devuelve JSON válido
    3. null explícito con razón documentada, nunca [] vacío ni valor inventado
    4. Idempotente — correr 2 veces da mismo resultado
    5. Testeable — cada campo es independiente

  Cadenas de prioridad:
    revenue, net_income, cfo, capex, fcf, sbc, deuda, cash, goodwill, shares
      → EDGAR primario (auditado, oficial SEC)
    operating_income (EBIT)
      → EDGAR.operating_income (OperatingIncomeLoss)
      → AV: revenue * margen_op (si ambos disponibles)
      → null
    margen_bruto, margen_op, margen_neto, roe, roa
      → Alpha Vantage (TTM ratios, más actualizados)
      → Calculado desde financials EDGAR
      → null
    beta, pe_forward, peg, ev_ebitda, pb
      → Alpha Vantage únicamente
      → null
    segmentos
      → EDGAR edgartools raw_segments_all (todos los ejes)
      → Limpieza: padres iterativa, geo vs producto, <3% → Otros
      → null explícito (nunca [])

FEATURES IMPLEMENTADAS:

  F-BATCH-1: Resolución automática de exchange sin sufijo
    Arquitectura: usuario escribe símbolo puro ("601899", "GRUMAB", "S68")
    Función resolve_yf_ticker(symbol, country) detecta el exchange:
      - Números 4 dígitos              → .HK (Hong Kong)
      - Números 5-6 dígitos (0/2/3)   → .SZ (Shenzhen)
      - Números 5-6 dígitos (6/9)     → .SS (Shanghai)
      - Country code "MX"             → .MX (BMV México)
      - Country code "ES"             → .MC (Bolsa Madrid)
      - Country code "SG"             → .SI (SGX Singapur)
      - Country code "FR"             → .PA (Euronext París)
      - Country code ""/"US"          → sin sufijo (NYSE/OTC ADR)
    TICKERS como lista de tuplas (símbolo, country_code)
    Rationale: usuarios nunca escriben sufijos de exchange manualmente

  F-BATCH-2: Skip automático de tickers ya procesados
    Función already_done(symbol) escanea CORRIDA_*_FEROLDI/
    buscando Diagrama_Sankey_{symbol}_*.html
    NOTA: no funcionó en corridas de esta sesión por restricción macOS TCC
    que impide glob cross-directory desde subproceso de Kika

  F-BATCH-3: Parsing de stdout de feroldi_sankey para ubicar HTML
    Problema: find_html() con glob fallaba sistemáticamente
    Causa raíz: la función dependía de ventana de 15 min que expiraba,
    y el path de búsqueda no siempre coincidía con el ticker resuelto
    Fix: feroldi_sankey imprime "Archivo: /path/to/html"
    El batch parsea esa línea directamente → infalible
    Fallback: find_html() como segundo intento

  F-BATCH-4: Detección correcta de sankey sin datos
    Problema: feroldi_sankey.py retorna exit code 0 incluso cuando
    imprime "❌ Error: no hay Revenue en el JSON"
    Fix: verificar que stdout contenga "✅ SANKEY GENERADO"
    Si no → marcar como error "sankey_no_data" en lugar de ok_no_html

  F-BATCH-5: Auto-incremento de número de corrida
    next_corrida_number() busca primer N libre en ~/Downloads/CORRIDA_N_FEROLDI/

  F-BATCH-6: Reporte detallado con errores agrupados por step
    Incluye yf_ticker resuelto, precio, status, html path, error detail

CAMBIOS:

  [F1 — OPTION C] Resolución de tickers numéricos (HK/China) — feroldi_recolectar.py
    · Nueva función _is_numeric_ticker(): detecta tickers puramente dígitos
    · Nueva función _resolve_edgar_company(): reemplaza Company(ticker) directo
      para tickers numéricos. Flujo:
        1. Detectar ticker numérico (solo dígitos)
        2. Obtener longName real via yfinance (con sufijos .HK, .SS, .SZ)
        3. Buscar en EDGAR por nombre con edgar_find()
        4. Validar: score ≥ 80 AND similitud difflib ≥ 0.50 (evita falsos positivos:
           "Industrial and Commercial Bank of China" → "Canadian Imperial Bank of
           Commerce" tenía score=60.3 → rechazado correctamente)
        5. Si match válido → usar ese CIK (ej: Prudential plc score=100, sim=1.00)
        6. Si no hay match → CIK=N/D, nombre desde yfinance, NO llamar a
           Company(ticker) que daría datos de empresa equivocada
    · get_edgar_edgartools() y extract_segments_edgartools() usan la nueva función
    · Impacto validado (09-05-2026):
        1398 (ICBC): antes "Entity 1398" datos US random →
                     ahora "Industrial and Commercial Bank of China Limited"
                     revenue $86B desde yfinance (nombre y datos correctos)
        2378 (Prudential plc): antes "Entity 2378" →
                     ahora CIK=0001116578 (EDGAR real), 3 segmentos detectados ✅
        0883/2318/0805: CIK=N/D, nombre correcto desde yfinance

  [F2 — 40-F] Soporte para filers canadienses — feroldi_recolectar.py
    · extract_segments_edgartools() reemplaza la cadena secuencial de ifs por
      un loop sobre todos los tipos de filing en orden de prioridad:
        10-K (no-amend) → 10-K → 20-F (no-amend) → 20-F → 40-F (no-amend) → 40-F
    · FIX CRÍTICO en la lógica: antes, si se encontraba un filing SIN XBRL
      (ej: BN/Brookfield tiene 20-F de 2002 sin XBRL) se retornaba None inmediatamente
      sin intentar el 40-F que sí tiene XBRL. Ahora: si xbrl() retorna None,
      se continúa con el siguiente tipo de filing.
    · Empresas canadienses afectadas: BN (Brookfield), BNS, CM, CNQ, AEM
    · Nota: BN tiene 7 segmentos en el 40-F pero bajo concepto no-estándar
      (ifrs-full:InterestRevenueExpense); el framework dimensional no los captura.
      Fix arquitectónico para ese concepto queda pendiente.

  [F3 — SANITY CHECK] Detección de CIK incorrecto — feroldi_recolectar.py
    · Después de asignar company_name desde EDGAR, si el nombre coincide con
      regex ^Entity\s+\d+$ → log 🚨 SANITY FAIL + cik_sanity_warning=True en JSON
    · No bloquea el pipeline: es un flag de auditoría para detección post-hoc
    · Con Fix 1 activo, este caso no debería ocurrir para tickers numéricos,
      pero protege contra otros escenarios donde edgartools devuelva nombres genéricos

  [F4 — ALREADY_DONE] Fix macOS TCC — feroldi_batch_corrida.py
    · already_done() usaba glob.glob() con wildcard de dos niveles:
        glob("Downloads/CORRIDA_*_FEROLDI/Diagrama_Sankey_TICKER_*.html")

    · Fix: reemplazar glob por enumeración explícita:
